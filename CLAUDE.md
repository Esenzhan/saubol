# SauBol — контекст проекта для Claude

Личный медицинский архив: загрузка документов → OCR → извлечение
биомаркеров через AI → медкарта с графиками → AI-чат по своим данным.
Стек, структура папок и деплой — см. `README.md`.

## Кому принадлежат данные

Реальные аккаунты на сайте — не для Дарии (владелицы репозитория), а для
члена её семьи. Подробный медицинский контекст (диагнозы, история болезни)
не хранится в этом файле и в репозитории намеренно — он живёт в памяти
Claude Code между сессиями. Если нужен этот контекст — просто спросите.

## Текущее состояние (на 2026-07-19)

Все 40 реальных медицинских документов пользователя загружены и полностью
обработаны вручную (без обращения к Anthropic API на извлечение — по
явной просьбе пользователя сэкономить токены при первичной загрузке); их
биомаркеры считаются подтверждёнными задним числом (`confirmed = true`),
без ретроактивного прогона через экран проверки ниже. Дальнейшая
автоматическая обработка новых загрузок использует AI как и задумано
изначально.

Архитектура документов:
- **Хранение файлов**: байты самого файла (`documents.file_data`, BYTEA)
  хранятся в Postgres/Neon, а не на диске backend — диск Render не
  переживает редеплой. Отдаются через `GET /api/documents/:id/file`.
- **Папки на странице «Документы»**: 4 фиксированных верхнеуровневых папки
  — Анализы / Приёмы врачей / Выписки и заключения / Другое, отрисованные
  как 2×2 сетка карточек с иконкой и своим акцентным цветом (без сортировки,
  по явному решению пользователя). Папку при загрузке выбирает
  пользователь (нажатием «+» внутри конкретной карточки) — `folder`
  передаётся в `POST /api/documents`, ИИ её не определяет. Список — дерево
  `FOLDER_TREE` в `frontend/src/documentFolders.js`.
- **Только «Анализы» сканируется на биомаркеры**: `analyzeDocument()` в
  `backend/services/ai.js` вызывается исключительно для документов,
  загруженных в «Анализы» — извлекает биомаркеры и придумывает короткое
  `display_name` **без даты** (дата показывается отдельно, см. ниже) +
  `document_date`. Конкретную подпапку (ОАК, Биохимия и т.п.) ИИ **не**
  определяет — это делает `classifyAnalysisSubfolder()` в
  `backend/services/folderClassifier.js`, правилами по ключевым словам в
  уже извлечённых именах биомаркеров (без обращения к модели), откалиб-
  рованными по всем реальным документам «Анализы» на сайте (бэктест:
  32/32 на 2026-07-19). Логика: каждый биомаркер относится максимум к
  одной категории (порядок правил важен — специфичные проверяются раньше
  широких); если категорий 2 — побеждает простое большинство, даже
  близкое к 50/50 (два родственных бланка вроде ОАК+ОАМ всё ещё
  осмысленно назвать по большему); если категорий 3+ — нужно явное
  доминирование (`DOMINANCE_THRESHOLD`, 70%) **и** отсутствие второй
  категории размером от `SECOND_CATEGORY_PANEL_SIZE` (5) показателей —
  иначе это реальная вторая панель (напр. гормоны рядом с ОАМ), а не шум
  классификации, и документ уходит в «Комплексные анализы». Список
  подпапок — `ANALYSIS_SUBFOLDERS` там же, держать в синхроне с
  `ANALYSIS_SUBFOLDERS` в `frontend/src/documentFolders.js` вручную.
  Две пойманные на реальных данных ловушки, обе всё ещё актуальны при
  правке правил: (1) JS `\b` не распознаёт границы слов вокруг кириллицы
  (основан на ASCII `\w`) — для коротких кириллических аббревиатур
  используйте `(?<![а-яё])слово(?![а-яё])`, а не `\bслово\b`, иначе оно
  тихо никогда не совпадёт; (2) ИИ добавляет суффикс "(моча)"/"мочи" к
  имени показателя не всегда, а только когда в одном документе рядом есть
  одноимённый показатель крови, который нужно отличить — чистый ОАМ без
  примесей может прийти с именами вида "Лейкоциты"/"Эритроциты
  неизмененные" без пометки, поэтому правило ОАМ также ловит
  специфичные-только-для-мочи термины (нитриты, слизь, цилиндры,
  эпителий, дрожжи, кристаллы, "относительная плотность",
  "неизменен(ные)") и проверяется раньше ОАК, а не только по суффиксу.
  Подпапки внутри «Анализы» сортируются по дате самого свежего документа
  (новые — первыми).
  Документы в остальных трёх папках проходят только OCR (текст сохраняется
  в `raw_text` для будущего ИИ-чата) — без вызова модели, без биомарке-
  ров, только хранение (см. `processOtherDocument` в `documents.js`).
- **Подтверждение биомаркеров**: свежеизвлечённые ИИ показатели пишутся с
  `biomarkers.confirmed = false` и не видны в медкарте/чате/списке имён
  (везде фильтр `confirmed = true`, см. `records.js`, `chat.js`) — только
  в самом документе, с жёлтым бейджем «Показатели ждут подтверждения» в
  папке «Анализы». Экран проверки (`ReviewPanel` в `Documents.jsx`)
  показывает извлечённые значения в редактируемых полях, поиск по
  каталогу исторических показателей (`GET /records/biomarkers/catalog`)
  для добавления пропущенных ИИ (единица измерения подтягивается из
  последней записи с тем же именем), и разом отправляет правки +
  добавления в `POST /api/documents/:id/review`, которая помечает всё
  как подтверждённое.
- **Биомаркеры**: `biomarkers.value` (числа) и `biomarkers.value_text`
  (качественные результаты вроде «не обнаружено») — оба поля важны и
  оба должны извлекаться, ничего не пропускать даже для отрицательных
  качественных находок (см. `sanitizeBiomarkers` в `ai.js`).
- **Защита от ошибок OCR**: `isImplausibleValue()` в `ai.js` помечает
  `flagged_for_review`, если результат физически невозможен (% > 100) или
  на порядок выходит за референсный диапазон — частый паттерн: OCR теряет
  десятичный разделитель («43,2%» → «432%»). Снимается при подтверждении
  на экране проверки.
- **Порядок показателей** в документе/на экране проверки — по `id`
  вставки (порядок извлечения ИИ, совпадает с порядком в оригинале), а не
  по алфавиту — так удобно сверять построчно с исходником. Обе выдачи
  (`GET /documents/:id`, `POST /documents/:id/review`) используют
  `ORDER BY id`, не `ORDER BY name`.
- **Две даты документа показаны нарочно наоборот интуиции**: серым текстом
  под названием — дата **из самого файла** (`document_date`); дата «от
  ...» в названии документа — дата **загрузки на сайт** (`created_at`).
  `display_name` в БД хранится без даты вообще — суффикс "от {дата
  загрузки}" добавляет `documentTitle()` в `frontend/src/documentDisplay.js`
  на лету, так он не может разъехаться с `created_at`. Там же
  `documentSecondaryDate()` форматирует `document_date` без создания
  `Date`-объекта: `document_date` — чистая дата без времени, а pg по
  умолчанию парсит Postgres `DATE` через локальную таймзону сервера в
  `Date`-объект, который при JSON-сериализации может сместиться на день —
  в `backend/db/pool.js` стоит `pg.types.setTypeParser(1082, v => v)`,
  отдаёт `document_date` как есть строкой `YYYY-MM-DD`; не убирайте этот
  парсер и не оборачивайте `document_date` в `new Date()` на фронтенде.
- **Дедупликация имён биомаркеров**: ИИ не всегда стабильно выбирает
  алфавит для визуально одинаковых букв (кириллическая «Е» vs латинская
  «E» — реальный кейс, «Иммуноглобулин Е»/«Иммуноглобулин E» существовали
  как два разных значения `name` и раздваивали историю в медкарте).
  `canonicalizeBiomarkerName()` в `backend/services/textNormalize.js`
  перед каждой вставкой сверяет новое имя с уже существующими у
  пользователя (сравнение без учёта регистра/пробелов/кириллично-латинских
  омоглифов) и подставляет уже устоявшееся написание вместо нового от ИИ —
  самокорректирующийся механизм, не завязан на конкретно этот случай.

Медкарта (`frontend/src/pages/MedCard.jsx`):
- График показателя показывает диапазон нормы (закрашенная область +
  линии), единицу измерения, флагованные точки — отдельным маркером.
- Список показателей сгруппирован по категориям (`biomarkerCategories.js`,
  независимый от `documentFolders.js` список — тоже держать в синхроне
  вручную при добавлении новых типов анализов).
- Показатели без числового значения (только `value_text`) рисуются списком
  по датам, а не графиком.
- Выбор показателя — не нативный `<select>` (на 126+ именах неудобно
  листать, поиска нет), а свой `BiomarkerPicker` в том же файле: кнопка
  открывает панель с полем поиска и тем же списком по категориям, закрытие
  по выбору или клику вне панели.
- **Приём лекарств** — отдельная таблица `medication_doses` (доза + дата),
  полностью независимая от `biomarkers`/`medcard_entries`. Никакого ИИ или
  OCR-извлечения дозировок нет и не планируется — единственный способ
  занести дозу это ручная форма под графиком «Приём лекарств» (или прямой
  импорт в БД для массового исторического забэкфилла, как было сделано для
  Метипред/Лефно/Цетрин по документу с ежедневной таблицей приёма). Своя
  карточка «Приём лекарств» ниже «Динамика показателя» с тем же
  `BiomarkerPicker` (переиспользован как есть, он не завязан на биомаркеры)
  показывает график по выбранному лекарству. На самом графике «Динамика
  показателя» есть отдельный `BiomarkerPicker` с `allowClear` — «Наложить
  приём лекарства» — накладывает вторую линию (доза) на правую ось Y.
  Совмещение точек показателя и дозы идёт по объединённому множеству дат
  (`combinedSeries` в `MedCard.jsx`): так как дозы почти всегда вносятся
  ежедневно, а анализы — редко, обычная категориальная ось с общими для
  обоих рядов данными точками избавляет от необходимости переводить ось X
  в числовую/временную шкалу. `connectNulls` на обеих линиях — иначе линия
  анализа рвётся на каждой дате, где есть только доза, и наоборот.

Календарь (`frontend/src/pages/Calendar.jsx`) + push-напоминания:
- `calendar_events` (дата + `remind_before_days`, по умолчанию 7) и
  `push_subscriptions` (endpoint/p256dh/auth) — обе таблицы только с ручным
  вводом, без ИИ.
- **Render free tier засыпает без входящих запросов** — напоминание не
  может жить на `setInterval` внутри самого backend-процесса, сервис
  может быть уснувшим ровно в нужный момент. Вместо этого
  `.github/workflows/calendar-reminders.yml` раз в день (05:00 UTC) дёргает
  `POST /api/cron/check-reminders` с секретом в заголовке `x-cron-secret`
  (`CRON_SECRET` — секрет GitHub Actions, должен буквально совпадать со
  значением на Render). Сам запрос попутно и «пробуждает» уснувший сервис.
- Условие в `routes/cron.js` — не «ровно N дней до события», а «окно
  напоминания уже наступило, событие не в прошлом, `notified_at IS NULL`»:
  один пропущенный день (сервис не ответил на пинг) не хоронит напоминание
  навсегда, оно уйдёт при следующем успешном запуске.
- Web Push нужна VAPID-пара: `VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY` на
  backend (Render) + тот же публичный ключ как `VITE_VAPID_PUBLIC_KEY` на
  frontend (Vercel, вшивается в бандл на этапе сборки, поэтому после
  добавления/смены нужен новый деплой). Ключи сгенерированы один раз через
  `web-push generate-vapid-keys` и живут в `.env`, не в репозитории.
- **iOS Safari**: `PushManager` недоступен в обычной вкладке — работает
  только после установки сайта на экран «Домой» (там `display: standalone`
  из `manifest.json` уже даёт этот путь). `PushBanner` в `Calendar.jsx`
  показывает это как заметку, а не прячет функцию молча.

Ассистент (`frontend/src/pages/Chat.jsx`): ответы модели — GFM-markdown
(заголовки, жирный текст, таблицы), рендерятся через `react-markdown` +
`remark-gfm` с компонентами под тему сайта, а не выводятся сырым текстом.
Сообщения пользователя остаются простым текстом. Таблицы получают свой
`overflow-x-auto` + класс `scroll-x` (см. про блокировку скролла ниже).

OCR (`backend/services/ocr.js`): перед каждым распознаванием Tesseract
изображение проходит предобработку через `sharp` — автоповорот по EXIF
(частая проблема с фото боком), градации серого, растяжение контраста,
шарпинг, увеличение до 2000px по ширине если меньше. Специально без
бинаризации (жёсткий чёрно-белый порог) — на фото с неровным освещением
или тенью это обычно портит результат сильнее, чем помогает. Касается
только сканов/фото без текстового слоя — цифровые PDF от КДЛ (почти все
реальные документы) идут через `pdf-parse` и не проходят через OCR вообще.

Производительность: все страницы (`Documents`, `Dashboard`, `MedCard`,
`Chat`, `Admin`) используют `swr` для кэша между переходами — при
возврате на страницу данные показываются мгновенно, без повторной
загрузки с сервера.

## Известные особенности при разработке (см. также git log)

- **Recharts**: не передавайте `dot` как голый JSX-элемент — только как
  функцию; `isAnimationActive={false}` на `<Line>` обязателен (иначе в
  автоматизированном браузере анимация может не успеть дорисоваться);
  не подавайте вручную посчитанный `domain` с некруглыми числами — только
  функции `(dataMin)=>...`.
- **Тестирование кликов/состояния в Browser pane**: после клика подождите
  ~100-150мс перед проверкой DOM/скриншотом — React не всегда успевает
  перерисовать синхронно, иначе можно поймать несуществующий баг.
- **`window.open()` после `await fetch`** блокируется мобильными браузерами
  — открывайте пустую вкладку синхронно в обработчике клика, а потом
  подставляйте туда URL после загрузки (см. `client.js#openDocumentFile`).
- **Локальный backend подключён к боевой Neon-базе** (`backend/.env`) —
  `npm run dev` в `backend/` работает с теми же данными, что и прод.
  Для UI-тестов заводите отдельный тестовый аккаунт через `/register` и
  удаляйте его после (`DELETE FROM users WHERE ...`, каскадом снесёт его
  документы/биомаркеры) — не трогайте реальные данные Есенжана напрямую.
- **`computer` key-действие в Browser pane ненадёжно бьёт Enter** в React-
  инпуты этого приложения (клавиша будто не долетает до обработчика) —
  если клик по `ref` не совпадает с ожидаемым элементом или Enter не
  срабатывает, дублируйте действие через `javascript_tool`
  (`element.dispatchEvent(new KeyboardEvent('keydown', {key:'Enter', bubbles:true}))`)
  вместо повторных попыток тем же способом.
- **Форма `/register` требует поле «Имя»** — если заполнить только
  email/пароль и сабмитнуть, бэкенд молча вернёт 500 без внятной причины
  на экране («Что-то пошло не так»). Заполняйте все три поля для
  тестовых аккаунтов.
- **Backend dev-сервер не поднимается сам** между заходами в Browser pane
  в рамках одной сессии — `preview_start({name:"frontend"})` не запускает
  `backend` попутно. Если запросы к `/api/...` вдруг стали 500/зависают,
  первым делом проверьте `preview_start({name:"backend"})`, а не ищите
  баг в коде.
- **Загрузка тестового файла в форму через `javascript_tool`**: не
  передавайте большой base64 прямо в вызов инструмента — на очень длинных
  строках он иногда приходит повреждённым и `atob()` падает с
  `InvalidCharacterError`. Надёжнее: положить файл в `frontend/public/`,
  затем в браузере `fetch('/имя.pdf')` → `.blob()` → `FormData` — так же
  быстро, но без риска порчи строки при передаче через инструмент.
- **`env(safe-area-inset-*)` как padding на элементе с уже заданной
  фиксированной шириной (`w-16`/`w-60` и т.п.) при `box-sizing:
  border-box`** (глобальный дефолт от Tailwind preflight) — отступ съедает
  место у контента изнутри вместо того, чтобы расширить блок, и всё
  визуально едет (см. `Layout.jsx`: сайдбар). Правильно — расширять саму
  ширину на ту же величину: `w-[calc(4rem+env(safe-area-inset-left))]` +
  `pl-[env(safe-area-inset-left)]`, тогда контентная часть всегда остаётся
  ровно `4rem` независимо от значения отступа (в т.ч. отступа 0).
- **Блокировка диагонального скролла**: `html, body { touch-action: pan-y
  }` в `index.css` — страница скроллится только вертикально, свайп по
  диагонали не тащит её вбок. Любой новый элемент с намеренным
  горизонтальным скроллом (широкая таблица и т.п.) должен явно получить
  класс `.scroll-x` (`touch-action: pan-x`), иначе внутри него скролл
  вбок тоже будет заблокирован унаследованным `pan-y`.
- **Нижнее мобильное меню**: не используйте одинаковые по ширине
  `flex-1`-колонки с текстом по центру для пунктов разной длины
  («Обзор» вместо «Настройки») — из-за разницы в длине слов расстояние
  от края экрана до текста визуально получается разным, хотя колонки
  равны. Правильно — `justify-between` с фиксированным `px` на самом
  контейнере `<nav>`, тогда оба крайних пункта отстоят от края экрана на
  одинаковую величину независимо от длины слова.
- **Состояние свёрнутости десктопного/landscape-сайдбара** (`Layout.jsx`)
  хранится в `localStorage["sidebarCollapsed"]` и переживает
  навигацию/перезагрузку. При сворачивании `<main>` расширяется сам
  (обычный flex-сосед), но не забывайте снимать/увеличивать `max-w-4xl` у
  `<main>` в свёрнутом состоянии — иначе контент (график в Медкарте)
  упрётся в кап и не займёт освободившееся место.

## Автономная работа

Разрешено самостоятельно коммитить и пушить новые фичи в `main` без
дополнительного подтверждения на каждый шаг (см. память Claude Code).
Перед любым визуальным редизайном — сначала показать 4-5 вариантов на
выбор, не внедрять сразу.
